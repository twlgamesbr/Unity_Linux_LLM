Why Asmdefs Matter

  • Isolated Compilation — Each asmdef compiles independently; a change in one does not force a recompile of all others.
  • Faster Domain Reload — Only the affected assembly is reloaded instead of the entire AppDomain, cutting iteration time significantly.
  • IL2CPP Tree-Shaking — Unity's IL2CPP can strip unused code at the asmdef granularity, producing smaller WebGL builds.
  • Clear Dependency Contract — Each assembly declares exactly what it needs; the compiler enforces boundaries.

Dependency Rules

  • One Asmdef Per Domain — Each logical domain (Runtime, Monitoring, Editor, Tests) gets its own .asmdef file.
  • No Circular References — asmdef dependencies must form a DAG. The compiler rejects cycles.
  • Monitoring Must NOT Reference Runtime — The monitoring assembly observes from outside the game loop. Referencing Runtime would couple telemetry to gameplay code, defeating isolation. (Shown as forbidden above.)
  • Unity Built-ins Are Auto-Referenced — UnityEngine, Netcode, InputSystem, etc. are marked autoReferenced and need no explicit asmdef reference.

WebGL Impact

  • 2×+ Binary Size — Without asmdefs, IL2CPP cannot strip dead code. Every MonoBehaviour, every serialized field ships to the browser.
  • Full Reload on Every Change — Editing any script triggers a full recompile of the entire Assembly-CSharp, which in WebGL means a full re-export in development.
  • Higher OOM Risk — WebGL browsers have limited heap (~1–2 GB). Bloating with unused code increases out-of-memory crashes, especially on mobile browsers.
  • No Granular Stripping — With asmdefs, you can mark monitoring classes as link.xml preserve-only or strip them entirely from WebGL builds.
