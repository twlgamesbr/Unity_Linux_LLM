## Main Load-Time Rule

Inactive GameObjects in the first scene can prevent Awake, Start, Update, network connection, LLM initialization, and UI setup from running, but they do not remove the serialized scene data from the initial .data payload. For real WebGL download and startup savings, move heavyweight content out of the first scene or out of the initial build payload.

Recommended boot shape:

1. First scene contains only EventSystem, AuthUI, PlayerAuthService, minimal network/auth bridge, loading status, and WebGLGameplayLoadController.
2. After successful login, load gameplay via additive scene or Addressables.
3. Instantiate or activate NPCDialogueSystem, NPC characters, nameplates, ChatCanvas, RAG, networking roots, inventories, and gameplay props only after login.
4. Move NPC prefabs, profile assets, character textures, audio, RAG files, and optional scenes into Addressables or AssetBundles when they are not required for login.
5. Keep LocalAI, Supabase, Qdrant, and Cognee clients lazy: create clients on first use, not scene Awake.

## First Scene Budget

The login scene should avoid:

- NPCDialogueManager.
- QdrantRAGService.
- LLMUnity components.
- LLMAgent components.
- RAG components.
- NPC prefabs or animated NPC roots.
- world-space nameplates.
- full ChatCanvas/dialogue history UI.
- AudioSources or VideoPlayers not needed for auth.
- large textures, meshes, animation controllers, materials, or StreamingAssets references.
- Resources.Load paths for gameplay prefabs.
- Supabase Realtime subscriptions before the user is authenticated and actually enters gameplay.

The login scene may keep:

- AuthUIController and PlayerAuthService.
- a tiny status/loading UI.
- AuthNetworkBridge if it does not start networking until auth success.
- WebGLGameplayLoadController.
- lightweight configuration ScriptableObjects only if they do not reference heavy assets.

## Deferred Content Strategy

- Use additive scene loading for complete gameplay areas when the whole gameplay scene can be loaded after auth.
- Use Addressables for individual prefabs, textures, profiles, knowledge files, and optional UI panels that should be downloaded only when requested.
- Avoid putting optional content in Resources, because Resources assets are included in the player build and are harder to reason about for payload size.
- StreamingAssets are copied to the build output. Large knowledge files there still increase download size unless they are hosted externally or moved to Addressables/remote bundles.
- AssetBundles/Addressables still need caching and versioning discipline; hash or catalog changes must be deployed together.
- Call Resources.UnloadUnusedAssets only at deliberate transition points, not during active dialogue.

## Player Settings For Release Builds

Use the checked-in build profile as authority. For release-oriented WebGL builds, verify:

- Compression Format: Brotli.
- Decompression Fallback: disabled when the server sends correct Content-Encoding and Content-Type.
- Data Caching: consider enabling for stable production deployments so repeat visits reuse Unity data in browser storage.
- WebAssembly 2023: enabled for modern browser target.
- IL2CPP Code Generation: Optimize Size.
- Managed Stripping Level: Medium or High only with link.xml and browser validation.
- Debug Symbols: off for production, on only for profiling/debugging.
- Exception Support: reduced for production only after stack traces are no longer needed.
- Analyze Build Size: enable during audits, not necessarily every build.
- Initial heap: keep only as large as needed for first scene; a high initial heap increases memory pressure.
- Maximum heap: keep enough headroom for gameplay but watch mobile browser limits.
- Threads: disabled unless the deployment sets cross-origin isolation headers and every target browser supports it.

## Server Headers And Hosting

Unity WebGL builds must be served over HTTP(S), not file://.

For Brotli output, the server must send:

- .wasm.br: Content-Encoding br, Content-Type application/wasm
- .js.br: Content-Encoding br, Content-Type application/javascript
- .data.br: Content-Encoding br, Content-Type application/octet-stream
- .json.br: Content-Encoding br, Content-Type application/json

For gzip output, use Content-Encoding gzip with the same content types.

If WebAssembly threads are enabled later, hosting also needs cross-origin isolation:

- Cross-Origin-Opener-Policy: same-origin
- Cross-Origin-Embedder-Policy: require-corp
- Cross-Origin-Resource-Policy: cross-origin or same-origin depending on asset hosting

Use immutable caching for hashed build artifacts. Avoid stale index.html caching unless the deploy process updates all referenced files atomically.

## Browser Networking Rules

- Web builds support UnityWebRequest for HTTP(S).
- Web builds can use WebSockets through supported networking packages.
- Raw .NET sockets and native socket access are not supported in browser WebGL.
- Browser localhost means the user's machine, not the deployment host. Replace local hostnames with the page host or configured backend host in WebGL.
- Use HTTPS for production and for browser-gated device APIs.
- Configure CORS on Supabase, LocalAI proxies, Qdrant/Cognee gateways, and any static asset host used by Addressables.
- Keep auth before realtime subscriptions. Delay Supabase Realtime until after successful login and gameplay entry.

## Browser Scripting And Native Plug-ins

- Put JavaScript interop in .jslib files and keep the surface small.
- Prefer UnityWebRequest, C# APIs, and package integrations before adding JS interop.
- Avoid high-frequency C# <-> JS calls in gameplay loops.
- Browser APIs often require a user gesture for fullscreen, pointer lock, clipboard, audio start, and some device access.
- Emscripten native plugins are Web-only and must be compiled with a compatible Emscripten version. Keep them isolated behind platform guards.

## Memory Guidance

- The Unity heap, asset data, loaded scenes, browser JS engine, decompression buffers, audio/video, and GPU resources all compete in browser memory.
- Reducing initial scene content reduces both initial data and peak startup memory.
- Large textures and meshes are common WebGL load blockers. Compress, downscale, split, and defer them.
- Use texture streaming and mip stripping only after visual QA.
- Avoid decompressed audio on load unless the clip is tiny and needed immediately.
- Do not allocate large transient strings or JSON documents in dialogue paths; stream, reuse buffers, and cap history.
- Keep NPC dialogue history bounded. Persist only the compact state needed after login.
- Use Task.Yield for WebGL-friendly async loops; avoid blocking waits.
- Schedule expensive cleanup at scene transitions.

## Graphics And Texture Rules

- Use WebGL 2 as the normal baseline.
- Keep render resolution independent from CSS canvas size when performance requires it.
- Reduce MSAA, shadows, HDR, post-processing, real-time lights, and expensive shader variants for WebGL.
- Strip shader variants through Graphics Settings only after testing all dialogue/NPC visual states.
- Choose texture compression by target:
  - Desktop browsers commonly support BC/DXT paths.
  - Mobile browsers commonly prefer ETC2 or ASTC depending on device support.
  - If one build targets both desktop and mobile, expect fallback or separate builds.
- Avoid runtime texture decompression submodules by importing textures in formats that target browsers can use directly.
- Use the Web Stripping Tool's Texture Decompression submodules only after profiling confirms the build never needs them.

## Audio And Video Rules

- Browser audio often needs a user gesture before playback.
- Keep login scene silent unless audio is essential.
- Use compressed audio for larger clips and avoid Decompress On Load for non-trivial files.
- Test looping clips in browser; Web audio timing differs from native platforms.
- Video playback support depends on browser codecs. Host video as external streamed files when possible instead of embedding large clips in the initial build.

## Input, Canvas, And Device APIs

- Fullscreen and cursor lock usually require user interaction.
- Keyboard focus depends on the canvas and surrounding page.
- Mobile keyboard, touch, sensors, webcam, microphone, and other device APIs can require HTTPS and permissions.
- Keep the canvas size responsive but cap render resolution for slow devices.
- For mobile WebGL, test orientation, touch, UI scaling, memory pressure, and background-tab resume.

## WebAssembly 2023

- WebAssembly 2023 can improve runtime performance and reduce code size for modern browsers.
- Keep it enabled for current desktop targets.
- If supporting older browsers, check Unity's WebAssembly 2023 prerequisites and fallback strategy before shipping.
- WebAssembly 2023 is separate from threads. Threads still require cross-origin isolation and browser support.

- A successful login alone is not enough; stripped-function errors can appear only after dialogue or networking paths run.

## Do Not Do

- Do not assume inactive scene objects reduce initial download size.
- Do not keep LLMUnity local model objects in the first WebGL scene.
- Do not initialize NPC dialogue, RAG, Cognee, Qdrant, or Realtime before login.
- Do not use raw .NET sockets in browser builds.
- Do not rely on localhost from a deployed WebGL page.
- Do not enable WebAssembly threads without server isolation headers.
- Do not turn on Optimize Code After Stripping for every diagnostic build.
- Do not manually edit .unity scenes; use GladeKit for scene changes.
- Do not strip modules based only on names. Profile and browser-test.

## Quick Question Routing

If asked "why is WebGL slow to load":

- Check initial scene payload and heavy references first.
- Check Brotli headers and decompression fallback second.
- Check initial heap, texture size, StreamingAssets, Resources, and embedded assets third.
- Check submodule stripping and build size last, because it is a release optimization after behavior is validated.

If asked "how do we defer NPC/dialogue":

- Use login-only scene.
- Load gameplay scene additively or via Addressables after auth.
- Initialize dialogue services after gameplay load.
- Subscribe Realtime only after dialogue is needed.
- Instantiate NPCs from Addressables or loaded gameplay scene, not from first-scene roots.

If asked "how do we validate":

- Build with the wrapper.
- Serve through HTTP with correct compressed headers.
- Test in Chromium/Firefox.
- Inspect browser console.
- Exercise login, networking, NPC spawn, first dialogue, NPC switch, and UI text.
- Compare network waterfall and first playable time before/after.
